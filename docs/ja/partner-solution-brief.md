# パートナーソリューション概要: FSx for ONTAP サーバーレス Observability

## 顧客の課題

Amazon FSx for NetApp ONTAP を運用するエンタープライズ顧客は、ファイルアクセス監査ログを既存の SIEM/Observability プラットフォームに配信する必要があります。現在のアプローチでは、syslog-ng やベンダー固有のフォワーダーを実行する EC2 インスタンスが必要であり、運用オーバーヘッド、パッチ適用負荷、固定月額コストが発生します。

**よくある顧客の質問**:
- 「EC2 を管理せずに FSx ONTAP 監査ログを Splunk/Datadog/Grafana に送るには？」
- 「ファイル共有上のランサムウェア活動をリアルタイムで検知できるか？」
- 「ファイルアクセス監査のコンプライアンス要件をどう満たすか？」

## ソリューション概要

Lambda + EventBridge + S3 Access Points を使用したサーバーレスログ配信パイプライン。EC2 インスタンスの管理は不要です。

```
FSx ONTAP → S3 Access Point → EventBridge Scheduler → Lambda → Vendor API
                                                                (Datadog/Splunk/Grafana/etc.)
```

包括的なカバレッジのための **3 つのイベントソース**:
1. **監査ログ**（S3 AP ポーリング）: ファイルアクセスイベント（誰が何にいつアクセスしたか）
2. **EMS Webhook**（API Gateway）: システムイベント（ARP ランサムウェア検知、クォータアラート）
3. **FPolicy**（ECS Fargate → SQS）: リアルタイムファイル操作ストリーム（サブ秒レイテンシ）

## ビジネス価値

| メトリクス | Before（EC2 ベース） | After（サーバーレス） |
|--------|--------------------|--------------------|
| 月額インフラコスト | ~$66（2x t3.medium + EBS） | ~$6（Lambda + EventBridge） |
| パッチ適用頻度 | 月次（OS + エージェント） | ゼロ（マネージドサービス） |
| スケーリング | 手動（インスタンスリサイズ） | 自動（Lambda 同時実行） |
| 初回ログ取得までの時間 | 数時間（プロビジョニング + 設定） | 30 分（CloudFormation デプロイ） |
| ベンダー切り替えコスト | 高（フォワーダー設定の書き直し） | 低（Lambda ターゲット変更または OTel Collector 使用） |

## ターゲット顧客プロファイル

### プロファイル A: Splunk モダナイゼーション
- **現状**: EC2 Universal Forwarder で FSx ONTAP ログを Splunk に配信
- **課題**: EC2 パッチ適用、エージェント更新、固定コスト
- **推奨パス**: Splunk サーバーレス統合 + EC2 移行ガイド
- **PoC スコープ**: サーバーレススタックを並行デプロイ、48 時間のイベント一致を比較

### プロファイル B: 新規 Observability プラットフォーム
- **現状**: FSx ONTAP 監査ログがどこにも配信されていない
- **課題**: ファイルアクセスパターンの可視性なし、コンプライアンスギャップ
- **推奨パス**: 無料枠ベンダーから開始（New Relic 100GB/月 または Grafana 50GB/月）
- **PoC スコープ**: 監査ポーラーをデプロイ、ログがクエリ可能なことを確認、最初のダッシュボード作成

### プロファイル C: マルチベンダー / ベンダーニュートラル
- **現状**: チーム間で複数の Observability ツールを使用
- **課題**: パイプラインの重複、ベンダーロックイン
- **推奨パス**: OTel Collector（単一 Lambda、複数バックエンドへのファンアウト）
- **PoC スコープ**: ローカル Docker 検証（5 分）、その後 2+ バックエンドで AWS デプロイ

### プロファイル D: セキュリティ / ランサムウェア検知
- **現状**: ファイルシステム異常に対するリアルタイムアラートなし
- **課題**: ランサムウェア検知が定期スキャンに依存
- **推奨パス**: EMS Webhook（ARP イベント）+ FPolicy（ファイル操作ストリーム）
- **PoC スコープ**: ARP トリガーをシミュレート、30 秒以内のアラートを確認

## PoC エンゲージメントモデル

### スコープ（1 週間のエンゲージメント）

| 日 | アクティビティ | 成果物 |
|-----|----------|-------------|
| 1 | 要件収集 + 環境アセスメント | PoC 計画書 |
| 2 | 前提条件デプロイ（S3 AP、監査ログ） | S3 AP 稼働 + 監査ログフロー確認 |
| 3 | ベンダー統合デプロイ + 配信検証 | バックエンドに最初のログ到着 |
| 4 | ダッシュボード + アラート + 運用検証 | ダッシュボードスクリーンショット + DLQ 空 |
| 5 | ドキュメント + Go/No-Go 判断 | PoC レポート + 次のステップ |

### 成功基準（Level 1 — 最低限）

- [ ] 1 件の監査ログレコードが Observability バックエンドに到着
- [ ] SSM チェックポイントが配信成功後に前進
- [ ] DLQ が空のまま（配信失敗ゼロ）
- [ ] CloudFormation のデプロイと削除がクリーンに完了
- [ ] 想定本番ボリュームに対するコスト見積もりを作成

### Go/No-Go 判断の入力

| 質問 | オーナー |
|----------|-------|
| 配信レイテンシは要件を満たすか？ | プラットフォームチーム |
| 本番ボリュームでのコストは許容範囲か？ | 財務 / FinOps |
| データ分類は外部配信を許可するか？ | セキュリティ / コンプライアンス |
| パイプラインの運用オーナーは誰か？ | 運用チーム |
| ベンダーのデータレジデンシーは許容可能か？ | 法務 / DPO |

## 差別化

| 比較対象 | 本ソリューション | 代替手段 |
|-----|--------------|-------------|
| EC2 ベース（AWS Blog） | EC2 ゼロ、従量課金、自動スケーリング | 固定コスト、パッチ適用必要 |
| CloudWatch Logs のみ | マルチベンダー、リッチクエリ、ダッシュボード | CloudWatch エコシステムに限定 |
| サードパーティエージェント | FSx にエージェント不要、S3 AP ネイティブ | エージェント互換性の問題 |
| 手動ログダウンロード | 自動化、リアルタイム、アラート | 手動、遅延、アラートなし |
