# パートナーソリューション概要: FSx for ONTAP サーバーレス Observability

🌐 **日本語**（このページ） | [English](../en/partner-solution-brief.md)

## 組織の課題

Amazon FSx for NetApp ONTAP を運用する組織は、ファイルアクセス監査ログを既存の SIEM/Observability プラットフォームに配信する必要があります。現在のアプローチでは、syslog-ng やベンダー固有のフォワーダーを実行する EC2 インスタンスが必要であり、運用オーバーヘッド、パッチ適用負荷、固定月額コストが発生します。

**よくある質問**:
- 「EC2 を管理せずに FSx for ONTAP 監査ログを Splunk/Datadog/Grafana に送るには？」
- 「ファイル共有上のランサムウェア活動をリアルタイムで検知できるか？」
- 「ファイルアクセス監査のコンプライアンス要件をどう満たすか？」

## ソリューション概要

Lambda + EventBridge + S3 Access Points を使用したサーバーレスログ配信パイプライン。EC2 インスタンスの管理は不要です。

```
FSx for ONTAP → S3 Access Point → EventBridge Scheduler → Lambda → Vendor API
                                                                (Datadog/Splunk/Grafana/etc.)
```

包括的なカバレッジのための **3 つのイベントソース**:
1. **監査ログ**（S3 AP ポーリング）: ファイルアクセスイベント（誰が何にいつアクセスしたか）
2. **EMS Webhook**（API Gateway）: システムイベント（ARP ランサムウェア検知、クォータアラート）
3. **FPolicy**（ECS Fargate → SQS）: リアルタイムファイル操作ストリーム（サブ秒レイテンシ）

## ビジネス価値

| メトリクス | Before（EC2 ベース） | After（サーバーレス） |
|--------|--------------------|--------------------|
| 月額インフラコスト | ~$66（2x t3.medium + EBS） | ~$6（Lambda + EventBridge） |
| パッチ適用頻度 | 月次（OS + エージェント） | ゼロ（マネージドサービス） |
| スケーリング | 手動（インスタンスリサイズ） | 自動（Lambda 同時実行） |
| 初回ログ取得までの時間 | 数時間（プロビジョニング + 設定） | 30 分（CloudFormation デプロイ） |
| ベンダー切り替えコスト | 高（フォワーダー設定の書き直し） | 低（Lambda ターゲット変更または OTel Collector 使用） |

## ターゲットユーザープロファイル

### プロファイル A: Splunk モダナイゼーション
- **現状** — EC2 Universal Forwarder で FSx for ONTAP ログを Splunk に配信
- **課題** — EC2 パッチ適用、エージェント更新、固定コスト
- **推奨パス** — Splunk サーバーレス統合 + EC2 移行ガイド
- **PoC スコープ** — サーバーレススタックを並行デプロイ、48 時間のイベント一致を比較

### プロファイル B: 新規 Observability プラットフォーム
- **現状** — FSx for ONTAP 監査ログがどこにも配信されていない
- **課題** — ファイルアクセスパターンの可視性なし、コンプライアンスギャップ
- **推奨パス** — 無料枠ベンダーから開始（New Relic 100GB/月 または Grafana 50GB/月）
- **PoC スコープ** — 監査ポーラーをデプロイ、ログがクエリ可能なことを確認、最初のダッシュボード作成

### プロファイル C: マルチベンダー / ベンダーニュートラル
- **現状** — チーム間で複数の Observability ツールを使用
- **課題** — パイプラインの重複、ベンダーロックイン
- **推奨パス** — OTel Collector（単一 Lambda、複数バックエンドへのファンアウト）
- **PoC スコープ** — ローカル Docker 検証（5 分）、その後 2+ バックエンドで AWS デプロイ

### プロファイル D: セキュリティ / ランサムウェア検知
- **現状** — ファイルシステム異常に対するリアルタイムアラートなし
- **課題** — ランサムウェア検知が定期スキャンに依存
- **推奨パス** — EMS Webhook（ARP イベント）+ FPolicy（ファイル操作ストリーム）
- **PoC スコープ** — ARP トリガーをシミュレート、30 秒以内のアラートを確認

### プロファイル E: CrowdStrike Falcon LogScale 統合
- **現状** — エンドポイント保護に CrowdStrike Falcon を導入済み。LogScale は利用可能だがファイルストレージのログは未取り込み
- **課題** — ファイルアクセス監査ログが Falcon 内でエンドポイントテレメトリと相関できていない
- **推奨パス** — Lambda → LogScale HEC（Splunk 互換エンドポイント、移行の手間ゼロ）
- **PoC スコープ** — 監査ログシッパーをデプロイ、LogScale リポジトリで構造化イベントを確認、最初の検索クエリを作成

## PoC エンゲージメントモデル

### スコープ（1 週間のエンゲージメント）

| 日 | アクティビティ | 成果物 |
|-----|----------|-------------|
| 1 | 要件収集 + 環境アセスメント | PoC 計画書 |

### 成功基準（Level 1 — 最低限）

- [ ] 1 件の監査ログレコードが Observability バックエンドに到着
- [ ] SSM チェックポイントが配信成功後に前進
- [ ] DLQ が空のまま（配信失敗ゼロ）
- [ ] CloudFormation のデプロイと削除がクリーンに完了
- [ ] 想定本番ボリュームに対するコスト見積もりを作成

## アプローチ比較

本サーバーレスパターンは複数ある選択肢の一つです。各アプローチはそれぞれ適した状況が異なり、最適な選択はチームの運用モデル・ログ量・既存投資によって変わります。トレードオフは本パターン自身も含めて対称的に記載します。

| アプローチ | 適した状況 | 考慮すべきトレードオフ |
|-----------|-----------|----------------------|
| 本サーバーレスパターン（Lambda / Firehose） | 変動的〜中小規模のログ量、インフラ管理を最小化したいチーム、マルチベンダー配信 | コールドスタートや呼び出しあたりの制限あり。継続的な超大量処理では Firehose やストリーミング設計が必要になる場合あり |
| EC2 ベース（例: AWS Blog の syslog-ng + Universal Forwarder） | 既存の EC2/フォワーダー投資、継続的な超大量スループット、ホストの完全制御 | ログ量に関わらず固定コスト。OS・エージェントのパッチ適用と容量管理はユーザー責任 |
| CloudWatch Logs のみ | AWS ネイティブのコンソール・ツールに標準化しているチーム | クエリ・ダッシュボードは CloudWatch エコシステム内に留まる。クロスベンダー配信には追加対応が必要 |
| ホスト常駐型ベンダーエージェント | 既にベンダーエージェント群を運用している環境 | エージェントはデータの所在で稼働する必要がある。FSx for ONTAP はサードパーティエージェントを内包しないため、ホスト型ソース向き |
| 手動ログダウンロード | 単発調査やアドホックな監査 | 自動化・スケジューリング・アラートなし。継続的なパイプラインには不向き |

### 選び方

- 運用負荷を抑えつつ変動的なログ量に対応したい → 本サーバーレスパターン
- 既に EC2 フォワーダーへ投資済み、または継続的な超大量スループットが必要 → EC2 ベース
- AWS ネイティブツールに完全に留まりたい → CloudWatch Logs
- ホスト型ソース向けにベンダーエージェント群へ標準化済み → ベンダーエージェント

本ソリューションはこれらの選択肢を置き換えることを目的とせず、状況がサーバーレスに適するチームに向けた代替手段を提供します。

## セットアップ所要時間リファレンス

| コンポーネント | デプロイ所要時間 | 前提条件 |
|-------------|---------------|---------|
| サーバーレスパイプライン（CloudFormation） | 30 分 | AWS アカウント、監査ログ有効化済み FSx for ONTAP |
| NetApp Console + System Manager（GUI） | 1-2 営業日 | NSS アカウント作成（無料、承認に 1 営業日） |
| EMS Webhook（クォータアラート） | 1 時間 | ONTAP CLI アクセス（fsxadmin） |
| FPolicy（リアルタイムファイル操作） | 2-4 時間 | VPC ネットワーキング、ECS Fargate、ONTAP CLI |
| フル PoC（パイプライン + ダッシュボード + アラート） | 1 週間 | 上記全て + ベンダーアカウント |

> **注意**
>
> NetApp Console アカウント作成にはエンドユーザーレベルのアクセス承認に 1 営業日が必要です（NetApp の登録 UI ではこの階層を「Customer/End User」と表記しています。詳細は [system-manager-gui-guide.md](system-manager-gui-guide.md) を参照）。GUI ベースの管理デモを含む PoC エンゲージメントのスケジュール時に考慮してください。
