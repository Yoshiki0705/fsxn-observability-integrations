# EC2 ベースパターンとサーバーレスパターンの比較

## 概要

本ドキュメントでは、FSx for NetApp ONTAP の監査ログを Splunk に配信する 2 つのアーキテクチャパターンを比較します。

- **EC2 ベースパターン**: EC2 インスタンス上の syslog-ng + Splunk Universal Forwarder
- **サーバーレスパターン**: S3 Access Point + EventBridge + Lambda + Secrets Manager

EC2 ベースパターンは AWS 公式ブログで紹介されているアーキテクチャです:
https://aws.amazon.com/jp/blogs/news/auditing-user-and-administrative-actions-on-amazon-fsx-for-netapp-ontap-using-splunk/

## アーキテクチャ図

### EC2 ベースパターン

```
┌─────────────────┐     ┌──────────────────┐     ┌─────────────────┐
│  FSx for ONTAP  │────▶│  EC2 Instance    │────▶│  Splunk Cloud   │
│  (監査ログ)      │     │  (t3.medium)     │     │  (HEC Endpoint) │
└─────────────────┘     └──────────────────┘     └─────────────────┘
                               │    │
                               │    │
                        ┌──────┘    └──────┐
                        ▼                  ▼
               ┌──────────────┐   ┌──────────────────┐
               │   syslog-ng  │   │  Splunk Universal │
               │  (ログ収集)   │   │  Forwarder (転送)  │
               └──────────────┘   └──────────────────┘
```

**コンポーネント構成:**
- EC2 インスタンス (t3.medium): syslog-ng と Universal Forwarder をホスト
- syslog-ng: FSx ONTAP からの syslog メッセージを受信・パース
- Splunk Universal Forwarder: パースされたログを Splunk Cloud/Enterprise へ転送

### サーバーレスパターン

```
┌─────────────────┐     ┌──────────────────┐     ┌─────────────────┐
│  FSx for ONTAP  │────▶│  FSx ONTAP       │────▶│  EventBridge    │
│  (監査ログ)      │     │  S3 Access Point │     │  Scheduler      │
└─────────────────┘     └──────────────────┘     └────────┬────────┘
                                                           │
                                                           ▼
┌─────────────────┐     ┌──────────────────┐     ┌─────────────────┐
│  Splunk Cloud   │◀────│     Lambda       │◀────│  Secrets Manager│
│  (HEC Endpoint) │     │  (ログ変換・配信)  │     │  (HEC Token)    │
└─────────────────┘     └──────────────────┘     └─────────────────┘
```

**コンポーネント構成:**
- S3 Access Point: FSx ONTAP 監査ログへの S3 API アクセスレイヤー
- EventBridge Scheduler: Lambda を定期起動（5分間隔）
- Lambda: ログ取得・パース・Splunk HEC へ配信
- Secrets Manager: HEC トークンの安全な管理

## 比較表

| 比較項目 | EC2 ベースパターン | サーバーレスパターン |
|---------|-------------------|-------------------|
| **月額コスト (1GB/日)** | ~$50–70/月 | ~$5–15/月 |
| **運用オーバーヘッド** | 高（3 コンポーネント管理） | 低（マネージドサービス） |
| **スケーリング** | 手動（インスタンスサイズ変更） | 自動（Lambda 同時実行） |
| **ログ配信レイテンシ** | 数秒〜数十秒 | 数分（Scheduler 間隔依存） |
| **高可用性** | 手動構成（Multi-AZ, ASG） | 組み込み（Lambda HA） |
| **パッチ管理** | ユーザー責任（OS, syslog-ng, UF） | AWS 責任（Lambda ランタイム） |

## コスト内訳 (1GB/日, ap-northeast-1)

### EC2 ベースパターン: ~$50–70/月

| AWS サービス | 月額コスト (USD) | 備考 |
|-------------|-----------------|------|
| EC2 (t3.medium) | ~$40 | オンデマンド、24/7 稼働 |
| EBS (gp3, 30GB) | ~$5 | OS + ログバッファ |
| データ転送 (アウトバウンド) | ~$5–10 | Splunk Cloud への転送 |
| Splunk Universal Forwarder | $0 | ライセンス無料 |
| CloudWatch (基本監視) | ~$0–5 | メトリクス + ログ |
| **合計** | **~$50–70** | |

### サーバーレスパターン: ~$5–15/月

| AWS サービス | 月額コスト (USD) | 備考 |
|-------------|-----------------|------|
| Lambda | ~$1–3 | 5分間隔実行、256MB、~30秒/回 |
| S3 リクエスト | ~$1 | ListObjects + GetObject |
| EventBridge Scheduler | ~$1 | 定期スケジュール |
| Secrets Manager | ~$1 | 1 シークレット + API コール |
| CloudWatch (ログ + メトリクス) | ~$1–5 | Lambda ログ + アラーム |
| データ転送 (アウトバウンド) | ~$0–5 | Splunk Cloud への転送 |
| **合計** | **~$5–15** | |

## 詳細比較

### 運用オーバーヘッド

| 項目 | EC2 ベース | サーバーレス |
|------|-----------|------------|
| 管理対象インフラ数 | 3（EC2, syslog-ng, UF） | 0（全てマネージド） |
| OS パッチ適用 | 月次（ユーザー責任） | 不要 |
| ログローテーション設定 | 必要（logrotate） | 不要 |
| セキュリティグループ管理 | 必要 | Lambda 用のみ |
| SSH キー管理 | 必要 | 不要 |
| ディスク容量監視 | 必要 | 不要 |

### スケーリング動作

| 項目 | EC2 ベース | サーバーレス |
|------|-----------|------------|
| 最大スループット | インスタンスサイズ依存 | Lambda 同時実行数依存（デフォルト 1000） |
| スケールアップ方式 | インスタンスタイプ変更（ダウンタイムあり） | 自動（設定変更不要） |
| スケールダウン | 手動 | 自動（使用量に応じた課金） |
| バースト対応 | 限定的 | Lambda 同時実行で対応 |

### ログ配信レイテンシ

| 項目 | EC2 ベース | サーバーレス |
|------|-----------|------------|
| ログ生成→収集 | 数秒（syslog リアルタイム） | 数分（Scheduler 間隔） |
| 収集→Splunk 到着 | 数秒（UF リアルタイム転送） | 数秒（Lambda→HEC） |
| エンドツーエンド | ~10–30 秒 | ~5–10 分 |

### 高可用性 (HA)

| 項目 | EC2 ベース | サーバーレス |
|------|-----------|------------|
| フェイルオーバー方式 | Auto Scaling Group + ALB | Lambda 組み込み HA |
| 復旧時間 | 数分（新インスタンス起動） | 即時（別 AZ で自動実行） |
| データ損失リスク | EBS 障害時にバッファ内ログ喪失 | DLQ でイベント保持 |
| 追加コスト | Multi-AZ で 2 倍 | 追加コストなし |

### パッチ管理

| 項目 | EC2 ベース | サーバーレス |
|------|-----------|------------|
| 責任者 | ユーザー | AWS（Lambda ランタイム） |
| 対象 | OS, syslog-ng, UF, Python 等 | なし（ユーザー管理不要） |
| 更新頻度 | 月次推奨 | AWS が自動適用 |
| ダウンタイム | パッチ適用時に発生 | なし |

## 推奨パターン選択ガイド

| シナリオ | 推奨パターン | 理由 |
|---------|------------|------|
| **低ボリューム (≤1GB/日)** | サーバーレス | コスト効率が圧倒的に高い（$5–15 vs $50–70）。運用負荷も最小 |
| **高ボリューム (>10GB/日)** | EC2 ベース（既存インフラあり）/ Firehose（新規） | 大量ログのリアルタイム処理に適する。既存 EC2 インフラがある場合は EC2 パターン、新規構築の場合は Firehose パターンを推奨 |
| **最小運用負荷** | サーバーレス | パッチ管理・スケーリング・HA が全て自動。インフラ管理ゼロ |

## まとめ

- **コスト**: サーバーレスパターンは EC2 ベースの約 1/5〜1/10 のコスト
- **運用**: サーバーレスパターンはインフラ管理が不要で運用負荷が大幅に低い
- **レイテンシ**: EC2 ベースはリアルタイムに近い配信が可能（数十秒）。サーバーレスは数分の遅延あり
- **スケーリング**: サーバーレスは自動スケーリング。EC2 は手動対応が必要
- **推奨**: 大多数のユースケース（特に 1GB/日以下）ではサーバーレスパターンを推奨

> **注意**: レイテンシ要件が厳しい場合（10 秒以内の配信が必須）は EC2 ベースパターンまたは EMS Webhook パスを検討してください。
