# 既存監査ツールとの共存ガイド

🌐 **日本語**（このページ） | [English](../en/existing-audit-tool-coexistence.md)

## 概要

本ドキュメントでは、サーバーレスオブザーバビリティパイプライン（Lambda + S3 Access Point + HEC/OTLP）と、環境内に既にデプロイされている既存のバッチベース監査ログツールとの共存方法を説明する。

サーバーレスパイプラインは、既存の監査レポートツールの**ドロップイン代替品ではない**。SOC ユースケース（リアルタイムアラート、SIEM 相関、XDR 統合）に最適化された補完的な検知・調査パスを提供する。既存ツールは引き続き、監査レポート、コンプライアンス証跡、内部調査ワークフローの役割を果たし得る。

---

## ONTAP 監査フォーマット制約

**重要**: ONTAP は SVM あたり1つの監査ログ出力フォーマットのみをサポートする。各 SVM の監査構成は EVTX または XML のいずれかを使用し、同時デュアルフォーマット出力はサポートされない。

参考: [NetApp KB — Can ONTAP generate CIFS audit logs in both EVTX and XML formats at the same time?](https://kb.netapp.com/on-prem/ontap/da/NAS/NAS-KBs/Can_ONTAP_generate_CIFS_audit_logs_in_both_EVTX_and_XML_formats_at_the_same_time)

| シナリオ | フォーマット | 利用者 |
|----------|--------|-----------|
| 既存ツールが EVTX を要求 | EVTX | 既存ツール + Windows Event Viewer |
| 本パイプライン（サーバーレス） | XML | Lambda パーサー（標準ライブラリ、バイナリ依存なし） |
| 同一 SVM で両方 | ❌ 同時使用不可 | フォーマットを選択するか、別 SVM を使用 |

### 共存戦略

| 戦略 | 説明 | トレードオフ |
|----------|-------------|------------|
| **A: SVM 分離** | EVTX 用 SVM（既存ツール）と XML 用 SVM（サーバーレスパイプライン）を分離。必要に応じて SnapMirror や qtree レベルのレプリケーションを使用。 | SVM 追加のオーバーヘッド、明確な所有権分離 |
| **B: フォーマット移行** | EVTX から XML へ移行。既存ツールが XML を処理できるか検証。 | 既存ツールが XML 未対応の可能性あり |
| **C: 単一フォーマット (EVTX) + Lambda EVTX パーサー** | EVTX フォーマットを維持。`fsxn_log_parser` の EVTX パースパスを使用（バイナリパーサーライブラリなしでは限定的なフィールド抽出）。 | `python-evtx` を Lambda Layer に含めないと機能制限あり |
| **D: 既存ツールからの正規化フィード** | 既存ツールを主要コレクターとして維持。既存ツールから正規化されたアクセスログを HEC またはファイルエクスポートで SIEM に送信。 | 既存ツールのエクスポート機能に依存 |
| **E: 両ツールで S3 Access Point 共有** | 両ツールが同じ S3 Access Point から読み取り（同一フォーマット）。同じフォーマットを処理できる場合のみ有効。 | フォーマット互換性が必要 |

## 所有権分割: レポーティング vs 検知

| 責務 | 既存監査ツール | サーバーレスパイプライン（本プロジェクト） |
|---------------|--------------------|------------------------------------|
| 監査レポート（月次、四半期） | ✅ 主担当 | ❌ 設計対象外 |
| コンプライアンス証跡エクスポート | ✅ 主担当 | ⚠️ SIEM エクスポートで対応可 |
| 内部調査（誰が・いつ・何を） | ✅ 主担当 | ✅ 補完（リアルタイム検索） |
| SOC 検知（大量削除、情報漏えい） | ❌ 設計対象外 | ✅ 主担当 |
| XDR/EDR 相関 | ❌ 不可 | ✅ 主担当（Falcon 等） |
| リアルタイムアラート（<5分） | ❌ バッチベース | ✅ 主担当 |
| 長期保存（アーカイブ） | ✅ 主担当 | ⚠️ SIEM 保持期間に依存 |
| ユーザー行動サマリー | ✅ 主担当 | ❌ 設計対象外 |
| 圧縮・ストレージ最適化 | ✅ 主担当 | ❌ SIEM に生イベント保存 |

## 重複収集リスク

既存ツールと本パイプラインが同じ監査ログを処理する場合:

### リスク

1. **監査レコードの不整合** — 一方のツールが先にファイルを処理し、ローテーション/削除された場合、もう一方がイベントを取りこぼす可能性
2. **二重ストレージコスト** — 同じイベントが2つのシステムに保存される
3. **タイムスタンプのずれ** — 異なる処理遅延により、クロスリファレンスが困難になる
4. **運用上の混乱** — 同じイベントに対して2つの「信頼できるソース」が存在

### 軽減策

- **明確な所有権モデル**: どのシステムがどのユースケース（レポーティング vs 検知）で権威あるかを定義
- **監査ボリュームの共有**: 両ツールが同じソース（S3 Access Point または NFS/SMB エクスポート）から読み取り — ソースレベルのデータ重複なし
- **タイムスタンプの一貫性**: 両ツールは取り込み時刻ではなく、元の ONTAP イベントタイムスタンプを保持すべき
- **ランブックの明確化**: どの調査シナリオでどのツールを使用するかを文書化

## 保持期間と証跡ポリシー

| 項目 | 既存ツール | サーバーレスパイプライン |
|--------|--------------|---------------------|
| 保持期間 | ツール設定による（コンプライアンスで1-7年が一般的） | SIEM 保持期間（ベンダー/契約による） |
| 証跡エクスポート | ツール固有のレポートフォーマット | SIEM エクスポート（JSON, CSV） |
| 証跡の連鎖（Chain of custody） | ツール内部で管理 | SIEM 監査証跡 + CloudTrail |
| 改ざん防止 | ツール固有（読み取り専用アーカイブ） | SIEM イミュータブルストレージ（ベンダーによる） |

**推奨**: 監査ログが規制上の証跡（ISMAP, FISC, SOC 2）の一部である場合、少なくとも1つのシステムが改ざん防止付き長期保存と文書化された証跡の連鎖を提供することを確認する。

## 移行時の検討事項

既存バッチツールから本サーバーレスパイプラインへ移行する場合:

1. **機能パリティの検証** — 本パイプラインは検知と検索を提供するが、サマリー化やコンプライアンスレポートテンプレートは提供しない
2. **フォーマット変更の影響検証** — EVTX から XML への切替は、その SVM の監査ログの全消費者に影響する
3. **並行運用期間の計画** — 検証期間中は両システムを並行運用する
4. **保持要件の確認** — 既存ツールのアーカイブを廃止する前に、SIEM 保持期間がコンプライアンス要件を満たすことを確認
5. **決定の文書化** — 移行承認理由、移管されるユースケース、残存ギャップを記録

## パートナー向けヒアリング項目

既存の監査ツールが存在する場合、以下を確認する:

- 現在デプロイされているツールは何か？（製品名、バージョン）
- 対象の SVM と共有は？
- 現在の監査ログフォーマットは？（EVTX or XML）
- 現在生成されているレポートは？（月次アクセス、削除追跡、機密フォルダ監視）
- 監査レポートとセキュリティ監視の所有者は？
- SIEM は既存ツールの代替、補完、または出力の消費のいずれを意図しているか？
- 監査証跡の保持要件は？
- 既存ツールは正規化フィード（CSV, JSON, syslog）を SIEM にエクスポートできるか？
