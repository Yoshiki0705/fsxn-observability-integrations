# セキュリティレビューチェックリスト

🌐 **日本語**（このページ） | [English](../en/security-review-checklist.md)

このチェックリストは、FSx for ONTAP オブザーバビリティパイプラインを本番環境や規制対象環境にデプロイする前に、セキュリティチームがレビューを行うためのものです。

## IAM ロールと権限

- [ ] Lambda 実行ロールが最小権限の原則に従っている
- [ ] S3 Access Point の List/GetObject リソースが特定の AP ARN にスコープされている
- [ ] Secrets Manager へのアクセスが特定のシークレット ARN にスコープされている（`*` ではない）
- [ ] SSM Parameter Store へのアクセスがチェックポイントパラメータパスにスコープされている
- [ ] SQS DLQ の権限が特定のキュー ARN にスコープされている
- [ ] データプレーンアクションに `Resource: "*"` を使用する IAM ポリシーがない
- [ ] CloudWatch Logs の権限が対象関数のロググループにスコープされている
- [ ] EventBridge Scheduler ロールが特定の Lambda 関数のみを呼び出せる

## シークレット管理

- [ ] API トークンやキーが CloudFormation パラメータにプレーンテキストで保存されていない
- [ ] インジェストトークン（ベンダー API キー）とプロビジョニング認証情報が分離されている
- [ ] Secrets Manager シークレットにローテーションスケジュールが定義されている
- [ ] Lambda がベンダーからの 401/403 をエラーログとして記録する（トークンを露出しない）
- [ ] トークンキャッシュの無効化がテストされている（コールドスタート時のリフレッシュ動作）
- [ ] 機密性のある CloudFormation パラメータに `NoEcho: true` が設定されている

## ネットワークセキュリティ

- [ ] Lambda の VPC 配置の判断が文書化され、正当性が示されている
  - VPC 外: S3 AP 読み取り専用関数向け（最もシンプル、NAT コスト不要）
  - VPC 内 + NAT: S3 AP と ONTAP REST API の両方が必要な関数向け
- [ ] API Gateway Webhook エンドポイントが本番環境で未認証のまま公開されていない
- [ ] Webhook に対してソース IP 制限、IAM 認証、Lambda オーソライザー、または WAF が設定されている
- [ ] ECS Fargate（FPolicy）の Security Groups が ONTAP IP からの TCP:9898 のみを許可している
- [ ] 正当な理由なく過度に許容的なエグレスルール（0.0.0.0/0）がない

## データ保護

- [ ] 監査ログが保存時に暗号化されている（ボリューム上の SSE-FSx）
- [ ] 転送中のデータが TLS を使用している（ベンダーエンドポイントへの HTTPS）
- [ ] DLQ メッセージに完全な監査ログペイロードが含まれていない（または保持期間が制限されている）
- [ ] CloudWatch Logs の暗号化が設定されている（必要に応じて KMS CMK）
- [ ] 監査イベントに含まれる情報を超える PII が Lambda 関数ログに記録されていない

## CloudFormation テンプレートセキュリティ

- [ ] テンプレートが `cfn-lint` 検証をエラーなしで通過する
- [ ] テンプレートが `cfn-guard` ポリシールールを通過する（設定されている場合）
- [ ] テンプレートにハードコードされた認証情報、アカウント ID、リソース ID がない
- [ ] パラメータに適切な `AllowedPattern` 制約が使用されている
- [ ] 重要なリソース（ログ、DLQ）に `DeletionPolicy: Retain` が設定されている
- [ ] 本番環境ではスタック終了保護が推奨されている

## 運用セキュリティ

- [ ] CloudWatch Alarms が以下に対して設定されている:
  - Lambda エラー（Errors メトリクス > 0）
  - DLQ メッセージ数（ApproximateNumberOfMessages > 0）
  - チェックポイントの陳腐化（カスタムメトリクスまたはスケジューラー失敗時のアラーム）
- [ ] ログ保持期間が明示的に設定されている（無期限ではない）
- [ ] DLQ アラームが運用チームへの通知をトリガーする
- [ ] デプロイ変更にプルリクエストレビューが必要である

## ベンダー送信先セキュリティ

- [ ] SaaS 送信先がセキュリティ/コンプライアンスチームによって承認されている
- [ ] データレジデンシー要件が満たされている（送信先リージョン）
- [ ] ベンダーのセキュリティ認証が確認されている（SOC 2、ISO 27001）
- [ ] ベンダーとのデータ処理契約（DPA）が締結されている
- [ ] ベンダー API エンドポイントが TLS 1.2+ を使用している
- [ ] リトライストーム時の認証情報露出を防ぐため、レート制限とバックオフが実装されている

## インシデント対応

- [ ] 配信失敗時のエスカレーションパスが定義されている
- [ ] DLQ リプレイ手順が文書化され承認されている
- [ ] トークン漏洩時の対応手順が存在する（ローテーション + 再デプロイ）
- [ ] パイプラインの健全性とセキュリティイベントの両方が監視対象である
