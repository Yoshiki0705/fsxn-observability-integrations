# コストモデル — Direct Send vs Collector vs Firehose

🌐 **日本語**（このページ） | [English](../en/cost-model.md)

## コストモデルの入力値

デプロイの月額コストを見積もるために、以下の値を収集してください:

| 入力値 | 測定方法 | 例 |
|-------|---------------|---------|
| 1 時間あたりの監査ファイル数 | S3 AP に対して `aws s3 ls` を 1 時間実行 | 50 ファイル/時間 |
| 平均ファイルサイズ | 100 ファイルをサンプリング | 200 KB |
| ファイルあたりのレコード数 | サンプルファイルをパース | 500 レコード |
| ファイルあたりの Lambda 実行時間 | CloudWatch `Duration` メトリクス | 800 ms |
| スケジュール間隔 | CloudFormation パラメータ | 5 分 |
| Grafana 取り込みボリューム | ファイル数/時間 × 平均サイズ | ~10 MB/時間 |
| Grafana 保持期間 | Grafana Cloud プラン | 30 日 |
| CloudTrail データイベント（有効時） | CloudTrail イベント数 | $0.10 / 100K イベント |
| Collector コンピュート（使用時） | ECS Fargate vCPU 時間 | 0.25 vCPU |
| NAT Gateway（VPC Lambda の場合） | 処理データ量 | $0.045/GB |

## パス比較

| パス | 固定コスト | 変動コスト | 最適なユースケース |
|------|-----------|---------------|----------|
| **Direct send**（Part 6） | ~$0 | Lambda 呼び出し + 実行時間 + Grafana 取り込み | 低〜中ボリューム、単一バックエンド |
| **OTel Collector**（Part 5） | Fargate/EC2 コンピュート | Lambda + Collector + マルチバックエンド取り込み | マルチバックエンド、エンリッチメント、秘匿化 |
| **Firehose**（Splunk/Datadog） | ~$0 | Firehose 取り込み（$0.029/GB）+ ベンダー取り込み | 大量ボリューム、組み込み Firehose 送信先あり |

## Direct Send コスト見積もり（例）

前提: 50 ファイル/時間、平均 200 KB、5 分ポーリング、ap-northeast-1

| コンポーネント | 計算 | 月額コスト |
|-----------|-------------|-------------|
| Lambda 呼び出し | 12/時間 × 720 時間 = 8,640 | ~$0.02 |
| Lambda 実行時間 | 8,640 × 4s 平均 × 256 MB | ~$0.15 |
| SSM Parameter Store | Standard tier、無料 | $0.00 |
| EventBridge Scheduler | 無料枠内 | $0.00 |
| SQS（DLQ） | 最小メッセージ数 | ~$0.00 |
| Secrets Manager | 1 シークレット × $0.40/月 | $0.40 |
| Grafana Cloud 取り込み | 50 × 200KB × 720h = ~7 GB/月 | プラン依存 |
| **AWS コスト合計** | | **~$0.57/月** |

> Grafana Cloud の取り込みコストはプランによって異なります。Free tier には 50 GB/月のログが含まれます。

## Direct Send が高コストになるケース

Direct Send のコストはボリュームに比例してスケールします。以下の場合に代替手段を検討してください:

- Lambda 実行時間がスケジュール間隔の 80% を超える（スループット上限）
- Grafana 取り込みがプラン割り当てを超過（超過料金発生）
- リトライ/重複配信が取り込みオーバーヘッドを大幅に増加
- 複数バックエンドが必要（Collector は送信先間でコンピュートを償却）

## Collector パス コスト見積もり（例）

前提: 同ボリューム、ECS Fargate 0.25 vCPU / 0.5 GB

| コンポーネント | 計算 | 月額コスト |
|-----------|-------------|-------------|
| Lambda（リーダー） | 上記と同じ | ~$0.17 |
| Fargate Collector | 0.25 vCPU × 720h × $0.04048 + 0.5 GB × 720h × $0.004445 | ~$8.90 |
| マルチバックエンド取り込み | Grafana + Datadog + アーカイブ | プラン依存 |
| **AWS コスト合計** | | **~$9.07/月** |

Collector パスは固定コストが高いですが、マルチバックエンド配信、エンリッチメント、永続キューイングを追加の Lambda 複雑性なしに実現できます。

## コスト最適化のヒント

- 緊急性の低い監査可視化には `rate(5 minutes)` 以上を使用
- `MAX_KEYS_PER_RUN` を Lambda メモリ/タイムアウト予算に合わせて設定
- Lambda `Duration` p95 を監視 — メモリ割り当てを適正化
- コンプライアンス要件に適した Grafana Cloud ログ保持期間を使用
- ほぼリアルタイムのトリガーが不要な場合、CloudTrail データイベントを避ける（$0.10/100K イベント）
- S3 AP アクセスの NAT Gateway コストを避けるため Lambda を VPC 外にデプロイ


## NetApp 固有のコストドライバー

FSx for ONTAP 環境では、以下の追加要因がパイプラインコストに影響します:

| ドライバー | 影響 | 制御方法 |
|--------|--------|----------------|
| 監査ログローテーション間隔 | ローテーション頻度が高い = ポールあたりのファイル数増加 = Lambda 実行時間増加 | ONTAP 監査設定でローテーション間隔を調整 |
| 平均監査ファイルサイズ | ファイルが大きい = 呼び出しあたりのパース時間増加 | MAX_KEYS_PER_RUN を適宜調整 |
| FPolicy イベントボリューム | 高ボリューム共有は毎秒数千イベントを生成可能 | ボリューム/パス/操作タイプで FPolicy スコープを制限 |
| EMS イベントボリューム | 通常は低い。インシデント時にスパイク | コスト面での対応不要 |
| FSx プロビジョンドスループット | S3 AP 読み取りスループットは FSx ファイルシステムスループットに制約される | 読み取りレイテンシを監視。必要に応じて FSx をスケール |
| Grafana 取り込み GB/日 | Grafana Cloud の主要な継続コスト | ポーリング間隔 + MAX_KEYS_PER_RUN で制御 |
| CloudTrail データイベント（有効時） | $0.10 / 100K イベント。大規模では顕著 | 代わりにポーリングパターンを使用（デフォルト） |
| NAT Gateway（VPC Lambda の場合） | $0.045/GB 処理 | S3 AP アクセスには Lambda を VPC 外にデプロイ |

### FPolicy ボリューム見積もり

以下のイベントレート範囲は一般的なワークロードパターンに基づく計画用の例であり、汎用的なサイジングガイダンスではありません。実際のイベントボリュームは SVM 設定、FPolicy スコープ、監視対象操作、クライアント動作、アクセスパターンに依存します。本番サイジングに使用する前に、自環境でイベントボリュームを測定してください。

FPolicy イベントボリュームはワークロードに依存します:

| ワークロードタイプ | 一般的なイベントレート | 推奨事項 |
|---------------|-------------------|----------------|
| ホームディレクトリ（低アクティビティ） | 10-100 イベント/分 | 全監視が実現可能 |
| 共有ファイルサーバー（中） | 100-1000 イベント/分 | セキュリティ関連操作にスコープを限定 |
| ビルド/CI 出力（高 I/O） | 1000+ イベント/分 | スコープを狭く限定、または監査ログを代用 |
| NAS 上のデータベースファイル | 非常に高い open/read/write | FPolicy スコープから除外 |


## 重複配信のコスト要因

Direct パスは at-least-once 配信を提供するため、部分的な成功後のリトライが取り込みボリュームを増加させる可能性があります。重複オーバーヘッドの見積もり:

```
effective_ingest = baseline_ingest × (1 + retry_duplicate_rate)
```

実際には、以下の条件で重複率は低い（< 1%）:
- Grafana Cloud OTLP Gateway が安定している（長時間の障害なし）
- Lambda タイムアウトが一般的な処理時間を十分に上回っている
- SSM チェックポイント更新が確実に成功している

重複率が増加するケース:
- Grafana エンドポイントに断続的な 5xx エラーがある（Lambda がリトライ後、次回実行で再処理）
- Lambda が送信後、チェックポイント更新前にタイムアウト
- 現在のチェックポイント状態を確認せずに Scheduler DLQ リプレイを実行
- アグレッシブなリトライポリシー（高い MaximumRetryAttempts）
- SSM Parameter Store の一時的な書き込み失敗

重複率が減少するケース:
- 大きな `SAFETY_THRESHOLD_MS`（タイムアウトの十分前に処理を停止）
- `auth_cache` の reload-on-401/403 が繰り返しの認証失敗を防止
- 安定した Grafana Cloud OTLP Gateway（長時間の障害なし）
- 条件付き書き込みを持つ DynamoDB オブジェクトレジャー（本番アップグレード）

コスト見積もりには、Direct パスの保守的な計画係数として 1〜3% の重複オーバーヘッドを使用してください。Collector/Alloy パスの永続キューではこれがほぼゼロに削減されます。
