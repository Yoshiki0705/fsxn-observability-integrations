# 移行ガイド: EC2 ベース Splunk → サーバーレス

## 概要

[EC2 ベースの Splunk 統合](https://aws.amazon.com/jp/blogs/news/auditing-user-and-administrative-actions-on-amazon-fsx-for-netapp-ontap-using-splunk/)（syslog-ng + Universal Forwarder）から、サーバーレス Lambda + HEC パターンへの移行ガイドです。

## アーキテクチャ比較

```
[移行前: EC2 ベース]
FSx for ONTAP → syslog-ng (EC2) → Splunk Universal Forwarder (EC2) → Splunk Enterprise/Cloud
  - 2台の EC2 インスタンスを管理
  - OS パッチ適用、エージェント更新が必要
  - ログ量に関係なく固定コスト

[移行後: サーバーレス]
FSx for ONTAP → S3 Access Point → EventBridge Scheduler → Lambda → Splunk HEC
  - EC2 インスタンスゼロ
  - パッチ適用不要、エージェント管理不要
  - 従量課金（ゼロスケール可能）
```

## 移行チェックリスト

### フェーズ 1: 準備（1日目）

- [ ] 現在の Splunk インデックスとソースタイプ設定を確認
- [ ] 現在の syslog-ng フィルタルール（転送対象イベント）を文書化
- [ ] Splunk HEC が有効でアクセス可能であることを確認
- [ ] サーバーレス統合用の新しい HEC トークンを作成
- [ ] FSx for ONTAP 監査ログが S3 に書き込まれていることを確認（S3 Access Point 経由）
- [ ] UF のカスタムフィールド抽出やトランスフォームを特定

### フェーズ 2: 並行デプロイ（2-3日目）

- [ ] 既存の EC2 パイプラインと並行してサーバーレススタックをデプロイ
- [ ] 検証用に**別の Splunk インデックス**（例: `fsxn_audit_serverless`）を使用
- [ ] 24-48時間、新旧パイプラインのイベントを比較
- [ ] フィールドマッピングの一致を確認（sourcetype, host, source フィールド）
- [ ] データ欠損がないことを確認（イベント数比較）

### フェーズ 3: 切り替え（4-5日目）

- [ ] サーバーレススタックを本番インデックス（`fsxn_audit`）に切り替え
- [ ] syslog-ng → UF パイプラインを無効化（サービス停止、インスタンスは未削除）
- [ ] EC2 インスタンスを停止（未終了）した状態で24時間監視
- [ ] ダッシュボードとアラートが引き続き機能することを確認

### フェーズ 4: クリーンアップ（7日目以降）

- [ ] EC2 インスタンスを終了（syslog-ng + UF）
- [ ] 関連するセキュリティグループ、IAM ロール、EBS ボリュームを削除
- [ ] ドキュメントとランブックを更新
- [ ] 旧 CloudFormation/Terraform リソースをアーカイブ

## フィールドマッピング

| EC2 UF フィールド | サーバーレス相当 | 備考 |
|-----------------|---------------|------|
| `host` | SVM 名（監査ログから） | 以前は EC2 ホスト名 |
| `source` | `fsxn-observability` | 以前は syslog パス |
| `sourcetype` | `fsxn:ontap:audit` | 同一（設定可能） |
| `index` | `fsxn_audit` | 同一（設定可能） |
| `_time` | 監査ログのイベントタイムスタンプ | 以前は syslog タイムスタンプ |

## Splunk ユーザーへの影響

### 変更なし
- インデックス名とソースタイプ（既存に合わせて設定可能）
- 検索クエリ（SPL）— 同じフィールド名
- ダッシュボードと保存済み検索
- アラートルール

### 変更あり
- `host` フィールド: EC2 ホスト名 → SVM 名
- `source` フィールド: syslog ファイルパス → `fsxn-observability`
- 配信レイテンシ: ほぼリアルタイム（syslog）→ ポーリング間隔（デフォルト5分）
- syslog メタデータ（facility, severity）なし — ONTAP イベントフィールドを使用

## コスト比較

| コンポーネント | EC2 ベース（月額） | サーバーレス（月額） |
|--------------|------------------|-------------------|
| EC2 インスタンス (2x t3.medium) | ~$60 | $0 |
| EBS ボリューム (2x 20GB) | ~$6 | $0 |
| Lambda | $0 | ~$5 (10 GB/月) |
| EventBridge Scheduler | $0 | ~$0.01 |
| Secrets Manager | $0 | ~$0.40 |
| **合計** | **~$66** | **~$6** |

> 削減効果: ~$60/月（約90%削減）、一般的な監査ログ量の場合。

## ロールバック計画

切り替え後に問題が発見された場合:

1. 停止した EC2 インスタンスを起動（syslog-ng + UF）
2. syslog-ng が FSx for ONTAP からイベントを受信していることを確認
3. サーバーレス CloudFormation スタックを削除
4. 問題を調査・解決してから再度移行を試行

## FAQ

**Q: 両方のパイプラインを同時に実行できますか？**
A: はい。検証中は別の Splunk インデックスを使用してください。両方とも同じ FSx for ONTAP 監査ログから読み取れます。

**Q: 切り替え時にイベントを失いますか？**
A: いいえ。サーバーレス Lambda はチェックポイントを使用しており、最後のチェックポイント以降のすべての監査ログファイルを処理します。重複期間中に一時的な重複イベントが発生する可能性があります。

**Q: ほぼリアルタイムの要件がある場合は？**
A: デフォルトのポーリング間隔は5分です。ほぼリアルタイムが必要な場合は、FPolicy パス（サブ秒レイテンシ）を検討するか、ポーリング間隔を1分に短縮してください。

**Q: Splunk ダッシュボードを変更する必要がありますか？**
A: 通常は不要です。ダッシュボードが `host=<ec2-hostname>` でフィルタしている場合は、`host=<svm-name>` に更新するかホストフィルタを削除してください。
