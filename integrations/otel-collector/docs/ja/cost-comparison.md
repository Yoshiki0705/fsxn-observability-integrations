# コスト比較: 直接送信 vs OTel Collector パス

## 概要

FSx for ONTAP 監査ログをバックエンドに配信する2つのアーキテクチャパターンのコスト比較です。

## パターン比較

### パターン A: Lambda → ベンダー API 直接送信

```
S3 → Lambda → Vendor API (Datadog/Splunk/etc.)
```

### パターン B: Lambda → OTel Collector → ベンダー

```
S3 → Lambda → OTel Collector → Vendor API(s)
```

## コスト要素

### パターン A: 直接送信

| コスト要素 | 単価 (ap-northeast-1) | 月間見積もり (100万イベント/月) |
|-----------|----------------------|-------------------------------|
| Lambda 実行 (256MB, 平均500ms) | $0.0000042/100ms | ~$10.50 |
| Lambda リクエスト | $0.20/100万 | ~$0.20 |
| S3 GetObject | $0.00037/1000 | ~$0.37 |
| データ転送 (外部) | $0.114/GB | ~$1.14 (1GB想定) |
| **合計** | | **~$12.21/月** |

### パターン B: OTel Collector パス

| コスト要素 | 単価 (ap-northeast-1) | 月間見積もり (100万イベント/月) |
|-----------|----------------------|-------------------------------|
| Lambda 実行 (256MB, 平均200ms) | $0.0000042/100ms | ~$4.20 |
| Lambda リクエスト | $0.20/100万 | ~$0.20 |
| S3 GetObject | $0.00037/1000 | ~$0.37 |
| ECS Fargate (0.5vCPU, 1GB, 24/7) | — | ~$15.00 |
| NAT Gateway (固定) | $0.062/時 | ~$45.00 |
| NAT Gateway (データ転送) | $0.062/GB | ~$0.06 (1GB想定) |
| **合計** | | **~$64.83/月** |

## コスト比較サマリー

| シナリオ | パターン A (直接) | パターン B (Collector) | 差額 |
|---------|-----------------|---------------------|------|
| 100万イベント/月 | ~$12 | ~$65 | +$53 |
| 1000万イベント/月 | ~$105 | ~$80 | -$25 |
| 1億イベント/月 | ~$1,050 | ~$200 | -$850 |

> **注意**: パターン B の固定費（Fargate + NAT Gateway）は ~$60/月。イベント量が増えるほどパターン B が有利になります。

## いつどちらを使うか

### パターン A（直接送信）を推奨するケース

- ✅ 単一ベンダーのみ使用
- ✅ 月間イベント量が少ない（< 500万）
- ✅ コスト最小化が最優先
- ✅ VPC 不要（Lambda は VPC 外で動作）
- ✅ シンプルなアーキテクチャを好む

### パターン B（OTel Collector）を推奨するケース

- ✅ マルチベンダー同時配信が必要
- ✅ ベンダー切り替えの可能性がある
- ✅ 月間イベント量が多い（> 1000万）
- ✅ バッファリング/リトライの高度な制御が必要
- ✅ ベンダーロックイン回避が重要
- ✅ 将来的にメトリクス/トレースも統合予定

## 隠れたコスト考慮事項

### パターン A の隠れたコスト

- **ベンダー切り替え時の開発コスト**: Lambda コードの書き換えが必要
- **マルチベンダー時の Lambda 重複**: 各ベンダー用に別 Lambda が必要
- **テスト/メンテナンスコスト**: ベンダー固有のコードが増える

### パターン B の隠れたコスト

- **運用複雑性**: Collector の監視・メンテナンスが追加
- **NAT Gateway**: VPC 内配置の場合の固定費
- **学習コスト**: OTel Collector の設定・運用知識

## コスト最適化のヒント

### パターン B のコスト削減

1. **NAT Gateway 不要な構成**: Lambda を VPC 外に配置し、Collector も VPC 外で動作させる（Docker on EC2 Spot）
2. **Fargate Spot**: 非クリティカルなログ処理には Fargate Spot を使用（最大70%割引）
3. **スケジュール停止**: 開発/テスト環境では夜間・週末に Collector を停止
4. **ARM64**: Graviton ベースの Fargate タスクで最大20%コスト削減

### パターン A のコスト削減

1. **Provisioned Concurrency 不使用**: コールドスタートを許容
2. **ARM64 Lambda**: Graviton ベースで最大20%コスト削減
3. **バッチ処理**: 複数ログファイルをまとめて処理
