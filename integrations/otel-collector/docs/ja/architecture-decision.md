# アーキテクチャ判断記録: Direct Send vs OTel Collector

## 判断マトリクス

| 評価軸 | Direct Send | OTel Collector |
|--------|-------------|----------------|
| **シンプルさ** | ✅ コンポーネント少、Lambda → Backend 直結 | ⚠️ 追加インフラの管理が必要 |
| **マルチバックエンド** | ❌ バックエンドごとに Lambda が必要 | ✅ 単一 OTLP ストリーム、Collector でファンアウト |
| **運用オーナーシップ** | Lambda チームが配信を所有 | プラットフォームチームが Collector を所有 |
| **バックエンド移行** | コード変更 + Lambda 再デプロイ | 設定変更のみ（コード変更不要） |
| **バックプレッシャー** | Lambda タイムアウト / リトライのみ | Collector キュー + リトライ + DLQ |
| **障害分離** | バックエンド障害が Lambda に影響 | バックエンド障害はエクスポーターに限定 |
| **認証情報管理** | 各 Lambda にバックエンド認証情報が必要 | Collector で認証情報を一元管理 |
| **設定ガバナンス** | Lambda ごとの環境変数 / Secrets | 一元化された YAML、CI で検証 |

## Direct Send を選択する場合

- バックエンドが単一で変更予定がない
- チームが Lambda とバックエンド運用の両方を所有
- 運用オーバーヘッドの最小化が最優先
- イベント量が少ない（< 100 events/min）
- ルーティング、フィルタリング、リダクションの要件がない
- バックエンドベンダーがネイティブ AWS 統合を提供（例: Firehose 宛先）

## OTel Collector を選択する場合

- 複数バックエンドが同一ログストリームを受信
- バックエンド移行が計画済みまたは可能性が高い
- 認証情報の一元管理が必要
- ルーティング/フィルタリングロジックをアプリケーションコードから分離したい
- プラットフォームチームが Collector を共有サービスとして提供
- コンプライアンスでリダクションや PII マスキングが必要
- バックプレッシャー処理と障害分離が重要

## 本番運用オーナーシップの質問

OTel Collector を本番環境にデプロイする前に、以下の質問に回答すること:

| # | 質問 | オーナー |
|---|------|----------|
| 1 | Collector の運用（デプロイ、パッチ、スケール）は誰が行うか？ | プラットフォーム / SRE チーム |
| 2 | 設定変更の承認フローは？ | PR レビュー → CI 検証 → 段階的ロールアウト |
| 3 | バックエンド間の障害分離はどう実現するか？ | エクスポーターごとのリトライキュー; 1 つの障害が他に影響しない |
| 4 | Collector 障害時の動作は？ | Lambda リトライ → DLQ → 復旧後に再処理 |
| 5 | 移行期間中のデュアルパス（Direct + Collector）のガバナンスは？ | 移行ランブックで明確なカットオーバー基準を定義 |

## プラットフォーム契約

### プロデューサー契約（Lambda / アプリケーション）

| ルール | 説明 |
|--------|------|
| OTLP のみ | すべてのプロデューサーは `http://<collector>:4318/v1/logs` に OTLP/HTTP で送信 |
| 承認済み属性 | スキーマ定義済み属性のみ使用（`service.name`, `event.type` 等） |
| バックエンド非認識 | プロデューサーはどのバックエンドがログを受信するか知らない |
| リトライポリシー | 5xx/タイムアウト時にリトライ、永続的失敗時は DLQ に送信 |

### プラットフォーム契約（Collector / インフラストラクチャ）

| ルール | 説明 |
|--------|------|
| Collector 設定 | プラットフォームチームが `otel-collector-config.yaml` を所有 |
| ルーティング判断 | ルーティングルールは Collector 設定で定義、アプリケーションコードではない |
| 認証情報注入 | Secrets Manager → コンテナ起動時に環境変数として注入 |
| SLA | Collector 可用性目標: 99.9%（health_check で監視） |
| 変更管理 | 設定変更は CI 検証 + 段階的ロールアウトが必須 |
