# エンタープライズワークロード補足資料

## SAP、VMware、ミッションクリティカル運用への適用

この OTel Collector パターンは、エンタープライズ運用チームが FSx for ONTAP テレメトリを複数の運用ドメインに配信する必要がある場合に適用されます:

- インフラストラクチャ / プラットフォーム運用
- セキュリティ運用（SOC/SIEM）
- アプリケーション運用（SAP Basis、データベース管理）
- マネージドサービスプロバイダー運用
- Observability ベンダー評価・移行

## このパターンが行うこと

- 正規化された監査イベント、EMS アラート、FPolicy シグナルを複数バックエンドに同時ルーティング
- テレメトリプロデューサー（Lambda）をバックエンド固有の配信から分離
- アプリケーションコード変更なしでバックエンド移行を実現
- 認証情報管理とルーティングポリシーを一元化

## このパターンが置き換えないもの

- 生の監査証跡保持（S3 Object Lock 付き EVTX/XML）
- ワークロードネイティブ監視（SAP Solution Manager、Oracle Enterprise Manager 等）
- アプリケーションレベルのインストルメンテーション（トレース、メトリクス）
- コンプライアンス証跡の管理チェーン

## FSx for ONTAP ワークロードコンテキスト

FSx for ONTAP は以下のエンタープライズワークロードをサポートします:
- SAP HANA および SAP アプリケーションサーバー
- Oracle Database および Oracle RAC
- Microsoft SQL Server
- VMware データストア（NFS 経由）
- 汎用 NAS（NFS/SMB）

これらのワークロードが生成する監査ログ、EMS、FPolicy テレメトリはすべて、本リポジトリで説明する OTel Collector パターンを通じてルーティングできます。

## エンタープライズ環境向け移行アプローチ

ミッションクリティカルワークロードでは、段階的アプローチを使用します:

1. **並行稼働**: Direct-send と Collector パスを同時に運用
2. **パリティ検証**: 少なくとも 1 回の完全な監査ローテーションサイクルでイベント数と重要フィールドを比較
3. **運用承認**: バックエンドクエリ、アラート、ランブックが Collector 配信データで機能することを確認
4. **設定凍結**: 本番カットオーバー前に Collector 設定をロック
5. **カットオーバー**: Direct-send パスを無効化
6. **観察期間**: 旧インフラストラクチャ削除前に 2 週間以上モニタリング
