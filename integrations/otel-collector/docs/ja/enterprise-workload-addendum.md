# エンタープライズワークロード補足

🌐 **日本語**（このページ） | [English](../en/enterprise-workload-addendum.md)

## SAP、VMware、ミッションクリティカル運用への適用

この OTel Collector パターンは、エンタープライズ運用チームが FSx for ONTAP のテレメトリを複数の運用ドメインに配信する必要がある場合に適用されます:

- インフラストラクチャ / プラットフォーム運用
- セキュリティ運用 (SOC/SIEM)
- アプリケーション運用 (SAP Basis、データベース管理)
- マネージドサービスプロバイダー運用
- Observability ベンダー評価・移行

## このパターンが提供するもの

- 正規化された監査イベント、EMS アラート、FPolicy シグナルを複数のバックエンドに同時ルーティング
- テレメトリプロデューサー (Lambda) をバックエンド固有の配信から分離
- アプリケーションコード変更なしでのバックエンド移行を実現
- 認証情報管理とルーティングポリシーの一元化

## このパターンが置き換えないもの

- 生の監査証跡保持 (S3 Object Lock による EVTX/XML)
- ワークロード固有の監視 (SAP Solution Manager、Oracle Enterprise Manager 等)
- アプリケーションレベルの計装 (トレース、メトリクス)
- コンプライアンス証跡の管理チェーン

## FSx for ONTAP ワークロードコンテキスト

FSx for ONTAP は以下のエンタープライズワークロードをサポートしています:
- SAP HANA および SAP アプリケーションサーバー
- Oracle Database および Oracle RAC
- Microsoft SQL Server
- VMware データストア (NFS 経由)
- 汎用 NAS (NFS/SMB)

これらのワークロードが生成する監査ログ、EMS、FPolicy テレメトリはすべて、本リポジトリで説明する OTel Collector パターンを通じてルーティングできます。

## エンタープライズ環境での移行アプローチ

ミッションクリティカルなワークロードでは、段階的なアプローチを使用します:

1. **並行稼働**: Direct-send パスと Collector パスを同時に運用
2. **パリティ検証**: 少なくとも 1 回の監査ログローテーションサイクル全体でイベント数と重要フィールドを比較
3. **運用承認**: バックエンドクエリ、アラート、ランブックが Collector 経由のデータで正常に動作することを確認
4. **設定凍結**: 本番切替前に Collector 設定をロック
5. **切替**: Direct-send パスを無効化
6. **観察期間**: 旧インフラ撤去前に 2 週間以上モニタリング
