# ONTAP テレメトリ信頼性の源泉

🌐 **日本語**（このページ） | [English](../en/ontap-telemetry-source-of-truth.md)

## テレメトリ権威性の階層

ONTAP は複数のテレメトリストリームを生成する。コンプライアンス、調査、運用において、どのソースが権威的であるかを理解することが重要。

```
┌─────────────────────────────────────────────────────────────────┐
│  Layer 1: Authoritative Evidence (Source of Truth)               │
│  ┌─────────────────────────────────────────────────────────┐    │
│  │  Raw ONTAP Audit Logs (EVTX/XML) in S3                  │    │
│  │  - Generated by ONTAP kernel                             │    │
│  │  - Immutable (S3 Object Lock + Versioning)               │    │
│  │  - Complete (no filtering, no sampling)                   │    │
│  │  - Timestamped by ONTAP system clock (NTP synced)        │    │
│  └─────────────────────────────────────────────────────────┘    │
├─────────────────────────────────────────────────────────────────┤
│  Layer 2: Operational Copies (OTel-Delivered)                    │
│  ┌─────────────────────────────────────────────────────────┐    │
│  │  Normalized OTLP logs in observability backends          │    │
│  │  - Parsed and enriched by Lambda                         │    │
│  │  - May be filtered, sampled, or redacted                 │    │
│  │  - Suitable for alerting, search, dashboards             │    │
│  │  - NOT sole compliance evidence                          │    │
│  └─────────────────────────────────────────────────────────┘    │
└─────────────────────────────────────────────────────────────────┘
```

## テレメトリソーステーブル

| ソース | 目的 | フォーマット | 権威性レベル | リテンションオーナー |
|--------|------|-------------|-------------|---------------------|
| **監査ログ** | ファイルアクセスコンプライアンス、調査 | EVTX / XML | 🔴 権威的 — 法的証拠 | コンプライアンスチーム |
| **EMS / ARP** | セキュリティイベント、ランサムウェア対策アラート | JSON (webhook) | 🟡 運用的 — セキュリティトリアージ | SecOps チーム |
| **FPolicy** | リアルタイムファイルアクティビティシグナル | バイナリプロトコル → JSON | 🟡 運用的 — ランサムウェア検出 | SecOps / SRE チーム |

### 権威性レベルの定義

| レベル | 意味 | ユースケース |
|--------|------|-------------|
| 🔴 権威的 | 法的に許容可能、不変、完全 | 監査対応、フォレンジック調査 |
| 🟡 運用的 | アクション可能、タイムリー、不完全な場合あり | アラート、トリアージ、ダッシュボード |

## 関係性: ONTAP テレメトリ → OTel Collector

OTel Collector は**配信レイヤー**であり、信頼性の源泉ではない。

```
┌──────────────────────────────────────────────────────────────────┐
│  ONTAP (Source)                                                    │
│  ┌────────────┐  ┌────────────┐  ┌────────────┐                  │
│  │ Audit Logs │  │  EMS/ARP   │  │  FPolicy   │                  │
│  │  (S3)      │  │ (Webhook)  │  │  (TCP)     │                  │
│  └─────┬──────┘  └─────┬──────┘  └─────┬──────┘                  │
└────────┼────────────────┼───────────────┼─────────────────────────┘
         │                │               │
         ▼                ▼               ▼
┌──────────────────────────────────────────────────────────────────┐
│  Ingestion Layer (Lambda)                                         │
│  - Parse EVTX/XML/JSON                                            │
│  - Normalize to OTLP schema                                       │
│  - Enrich with service.name, event.type                           │
└────────────────────────┬─────────────────────────────────────────┘
                         │ OTLP/HTTP
                         ▼
┌──────────────────────────────────────────────────────────────────┐
│  Distribution Layer (OTel Collector)                               │
│  - Route to multiple backends                                     │
│  - Filter / redact / batch                                        │
│  - Retry and queue management                                     │
│  - NO data generation — only forwards                             │
└───────┬──────────┬──────────┬──────────┬─────────────────────────┘
        ▼          ▼          ▼          ▼
    Datadog    Grafana   Honeycomb    SIEM
   (30 days)  (30 days)  (60 days)  (1 year)
```

## 監査ボリュームサイジング考慮事項

### 監査ログ量に影響する要因

| 要因 | 影響 | サイジングガイダンス |
|------|------|---------------------|
| SVM 数 | 線形 | 各 SVM が独立した監査ストリームを生成 |
| ファイル操作レート | 線形 | CIFS/NFS 操作数/秒 |
| 監査ポリシースコープ | 可変 | イベント増 = ボリューム増 |
| ログフォーマット | 約2倍の差 | EVTX は XML より小さい |
| ローテーション間隔 | ファイル数に影響 | 短い間隔 = ファイル数増、各ファイルは小さい |

### ボリューム見積もり

```bash
# SVM あたりの日次監査ボリューム見積もり
# 目安: 1 KB/監査イベント (XML)、0.5 KB/イベント (EVTX)
#
# 例: 1000 file ops/min × 60 min × 24 hr × 1 KB = ~1.4 GB/day (XML)
# 例: 1000 file ops/min × 60 min × 24 hr × 0.5 KB = ~0.7 GB/day (EVTX)

# 現在の監査ボリュームを確認
aws s3 ls s3://<audit-bucket>/audit/<svm-name>/ --recursive --summarize \
  | tail -2
```

### S3 ストレージコスト予測

| 日次ボリューム | 月額 (Standard) | 月額 (90日後 Glacier) | 7年間合計 |
|---------------|----------------|----------------------|----------|
| 1 GB/day | ~$0.70 | ~$0.30 (移行後) | ~$150 |
| 10 GB/day | ~$7.00 | ~$3.00 (移行後) | ~$1,500 |
| 100 GB/day | ~$70.00 | ~$30.00 (移行後) | ~$15,000 |

## 監査ローテーションと Collector タイムスタンプウィンドウの整合

### ONTAP 監査ログローテーション

ONTAP はサイズまたはスケジュールに基づいて監査ログをローテーションする:

```
# ONTAP audit rotation settings (example)
vserver audit modify -vserver <svm-name> \
  -rotate-size 100MB \
  -rotate-schedule-month - \
  -rotate-schedule-dayofweek - \
  -rotate-schedule-day - \
  -rotate-schedule-hour 0,6,12,18 \
  -rotate-schedule-minute 0
```

### Collector 処理との整合

| ONTAP ローテーション | Lambda スケジュール | 整合リスク | 緩和策 |
|---------------------|--------------------|-----------:|--------|
| 100 MB ごと | 5 分ごと | 低 — 頻繁なポーリングでローテーションを捕捉 | 最終処理ファイルをチェックポイント |
| 6 時間ごと | 5 分ごと | 低 — ローテーション後数分以内に処理 | チェックポイント + S3 イベント通知 |
| オンデマンド | 5 分ごと | 中 — 手動ローテーションが見逃される可能性 | ポーリングウィンドウがギャップをカバーすることを確認 |

### タイムスタンプウィンドウの考慮事項

```
Timeline:
  T0: ONTAP generates audit event (ONTAP timestamp)
  T1: ONTAP writes to audit log file (buffered)
  T2: Audit log rotated → S3 upload begins
  T3: S3 object available (S3 timestamp)
  T4: Lambda triggered (EventBridge/Schedule)
  T5: Lambda sends OTLP to Collector
  T6: Collector exports to backend (backend ingest timestamp)

  Typical latency: T0 → T6 = 1-10 minutes
  Maximum latency: T0 → T6 = rotation interval + processing time
```

**重要ルール**: バックエンドの取り込みタイムスタンプではなく、ONTAP が生成したタイムスタンプ（`event.timestamp`）を正規のイベント時刻として常に使用すること。

## イベント数の照合

### ソース vs バックエンドの比較

```bash
# Step 1: Count source events (S3 audit files for a date range)
SOURCE_COUNT=$(aws s3api list-objects-v2 \
  --bucket <audit-bucket> \
  --prefix "audit/<svm-name>/2026/01/" \
  --query "Contents[].Key" --output json | jq length)

echo "Source files: ${SOURCE_COUNT}"

# Step 2: Count events per file (requires parsing)
# Use the log-parser layer to count events in each EVTX/XML file
python3 -c "
from fsxn_log_parser.parser import parse_audit_log
import json, sys
# Count events in a sample file
events = parse_audit_log('sample-audit.evtx')
print(f'Events in file: {len(events)}')
"

# Step 3: Compare with backend event count
# Datadog: Use Log Analytics query
# Grafana/Loki: Use LogQL count query
# Honeycomb: Use count() query
```

### 照合ポリシー

| 頻度 | アクション | 許容差異 | エスカレーション |
|------|----------|--------:|-----------------|
| 日次（自動） | S3 ファイル数 vs バックエンドイベント数を比較 | ≤ 1% | 超過時にアラート |
| 週次（手動） | 差異トレンドのレビュー | ≤ 0.5% 累積 | 根本原因を調査 |
| 月次 | 完全照合レポート | 0% 未説明 | コンプライアンスチームレビュー |
| インシデント後 | Collector 障害または設定変更後 | ギャップを文書化 | DLQ/S3 から再処理 |

### 一般的な差異の原因

| 原因 | 方向 | 検出 | 解決 |
|------|------|------|------|
| Lambda リトライ → 重複 | バックエンド > ソース | 重複 event_id | バックエンドで重複排除 |
| Collector 障害 → 欠損 | バックエンド < ソース | タイムラインのギャップ | DLQ から再処理 |
| フィルタ/サンプリング適用 | バックエンド < ソース | 設定により想定内 | 照合レポートに文書化 |
| ローテーションタイミング | 一時的な不一致 | 1 サイクル内に解消 | 次のポーリングを待つ |

## Collector 設定変更の監査証跡

### 設定変更を追跡する理由

`otel-collector-config.yaml` への変更はすべて、どのイベントがどのバックエンドに到達するかに影響する。コンプライアンスのために以下を証明する必要がある:
- 任意の時点でどのルーティングがアクティブだったか
- 誰が変更を承認したか
- 変更中にイベントが失われたかどうか

### 必須監査フィールド

| フィールド | ソース | 例 |
|-----------|--------|-----|
| 誰が | Git author | `user <user@example.com>` |
| いつ | Git timestamp | `2026-01-15T10:30:00+09:00` |
| 何を | Git diff | `otlp_http/elastic` エクスポーターを追加 |
| なぜ | PR description | "セキュリティチーム用に Elastic バックエンドを追加" |
| 承認 | PR reviewer | `security-team-lead` |
| デプロイ | CI/CD timestamp | `2026-01-15T11:00:00+09:00` |

### 監査クエリコマンド

```bash
# Full history of Collector config changes
git log --format="%H %ai %an %s" -- \
  'integrations/otel-collector/otel-collector-config*.yaml' \
  > collector-config-audit.log

# Show what changed in a specific commit
git show <commit-hash> -- \
  'integrations/otel-collector/otel-collector-config*.yaml'

# Find who added/removed a specific exporter
git log -p --all -S 'otlp_http/elastic' -- \
  'integrations/otel-collector/otel-collector-config*.yaml'
```

### 設定変更影響評価

設定変更を適用する前に評価すること:

| 質問 | スキップした場合のリスク |
|------|------------------------|
| バックエンドの追加/削除か？ | イベントの損失または重複の可能性 |
| フィルタリングルールの変更か？ | コンプライアンスイベントがドロップされる可能性 |
| バッチ設定の変更か？ | レイテンシまたはスループットに影響 |
| リトライ設定の変更か？ | データ損失ウィンドウが拡大する可能性 |
| ロールバックはテスト済みか？ | 変更失敗時に障害が長期化 |
