# ONTAP Telemetry Source of Truth

## Telemetry Authority Hierarchy

ONTAP generates multiple telemetry streams. Understanding which source is authoritative for compliance, investigation, and operations is critical.

```
┌─────────────────────────────────────────────────────────────────┐
│  Layer 1: Authoritative Evidence (Source of Truth)               │
│  ┌─────────────────────────────────────────────────────────┐    │
│  │  Raw ONTAP Audit Logs (EVTX/XML) in S3                  │    │
│  │  - Generated by ONTAP kernel                             │    │
│  │  - Immutable (S3 Object Lock + Versioning)               │    │
│  │  - Complete (no filtering, no sampling)                   │    │
│  │  - Timestamped by ONTAP system clock (NTP synced)        │    │
│  └─────────────────────────────────────────────────────────┘    │
├─────────────────────────────────────────────────────────────────┤
│  Layer 2: Operational Copies (OTel-Delivered)                    │
│  ┌─────────────────────────────────────────────────────────┐    │
│  │  Normalized OTLP logs in observability backends          │    │
│  │  - Parsed and enriched by Lambda                         │    │
│  │  - May be filtered, sampled, or redacted                 │    │
│  │  - Suitable for alerting, search, dashboards             │    │
│  │  - NOT sole compliance evidence                          │    │
│  └─────────────────────────────────────────────────────────┘    │
└─────────────────────────────────────────────────────────────────┘
```

## Telemetry Source Table

| Source | Purpose | Format | Authority Level | Retention Owner |
|--------|---------|--------|-----------------|-----------------|
| **Audit Logs** | File access compliance, investigation | EVTX / XML | 🔴 Authoritative — legal evidence | Compliance team |
| **EMS / ARP** | Security events, anti-ransomware alerts | JSON (webhook) | 🟡 Operational — security triage | SecOps team |
| **FPolicy** | Real-time file activity signals | Binary protocol → JSON | 🟡 Operational — ransomware detection | SecOps / SRE team |

### Authority Level Definitions

| Level | Meaning | Use Case |
|-------|---------|----------|
| 🔴 Authoritative | Legally admissible, immutable, complete | Audit response, forensic investigation |
| 🟡 Operational | Actionable, timely, may be incomplete | Alerting, triage, dashboards |

## Relationship: ONTAP Telemetry → OTel Collector

The OTel Collector is a **distribution layer**, not a source of truth.

```
┌──────────────────────────────────────────────────────────────────┐
│  ONTAP (Source)                                                    │
│  ┌────────────┐  ┌────────────┐  ┌────────────┐                  │
│  │ Audit Logs │  │  EMS/ARP   │  │  FPolicy   │                  │
│  │  (S3)      │  │ (Webhook)  │  │  (TCP)     │                  │
│  └─────┬──────┘  └─────┬──────┘  └─────┬──────┘                  │
└────────┼────────────────┼───────────────┼─────────────────────────┘
         │                │               │
         ▼                ▼               ▼
┌──────────────────────────────────────────────────────────────────┐
│  Ingestion Layer (Lambda)                                         │
│  - Parse EVTX/XML/JSON                                            │
│  - Normalize to OTLP schema                                       │
│  - Enrich with service.name, event.type                           │
└────────────────────────┬─────────────────────────────────────────┘
                         │ OTLP/HTTP
                         ▼
┌──────────────────────────────────────────────────────────────────┐
│  Distribution Layer (OTel Collector)                               │
│  - Route to multiple backends                                     │
│  - Filter / redact / batch                                        │
│  - Retry and queue management                                     │

└───────┬──────────┬──────────┬──────────┬─────────────────────────┘
        ▼          ▼          ▼          ▼
    Datadog    Grafana   Honeycomb    SIEM
   (30 days)  (30 days)  (60 days)  (1 year)
```

## Audit Volume Sizing Considerations

### Factors Affecting Audit Log Volume

| Factor | Impact | Sizing Guidance |
|--------|--------|-----------------|
| Number of SVMs | Linear | Each SVM generates independent audit streams |
| File operation rate | Linear | CIFS/NFS operations per second |
| Audit policy scope | Variable | More events = more volume |
| Log format | ~2x difference | EVTX is more compact than XML |
| Rotation interval | Affects file count | Shorter intervals = more files, smaller each |

### Volume Estimation

```bash
# Estimate daily audit volume per SVM
# Typical: 1 KB per audit event (XML), 0.5 KB per event (EVTX)
#
# Example: 1000 file ops/min × 60 min × 24 hr × 1 KB = ~1.4 GB/day (XML)
# Example: 1000 file ops/min × 60 min × 24 hr × 0.5 KB = ~0.7 GB/day (EVTX)

# Check current audit volume
aws s3 ls s3://<audit-bucket>/audit/<svm-name>/ --recursive --summarize \
  | tail -2
```

## Audit Rotation and Collector Timestamp Window Alignment

### ONTAP Audit Log Rotation

ONTAP rotates audit logs based on size or schedule:

```
# ONTAP audit rotation settings (example)
vserver audit modify -vserver <svm-name> \
  -rotate-size 100MB \
  -rotate-schedule-month - \
  -rotate-schedule-dayofweek - \
  -rotate-schedule-day - \
  -rotate-schedule-hour 0,6,12,18 \
  -rotate-schedule-minute 0
```

### Alignment with Collector Processing

| ONTAP Rotation | Lambda Schedule | Alignment Risk | Mitigation |
|----------------|-----------------|----------------|------------|
| Every 100 MB | Every 5 min | Low — frequent polling catches rotations | Checkpoint last processed file |
| Every 6 hours | Every 5 min | Low — Lambda processes within minutes of rotation | Checkpoint + S3 event notification |
| On-demand | Every 5 min | Medium — manual rotation may be missed | Ensure polling window covers gap |

### Timestamp Window Considerations

```
Timeline:
  T0: ONTAP generates audit event (ONTAP timestamp)
  T1: ONTAP writes to audit log file (buffered)
  T2: Audit log rotated → S3 upload begins
  T3: S3 object available (S3 timestamp)
  T4: Lambda triggered (EventBridge/Schedule)
  T5: Lambda sends OTLP to Collector
  T6: Collector exports to backend (backend ingest timestamp)

  Typical latency: T0 → T6 = 1-10 minutes
  Maximum latency: T0 → T6 = rotation interval + processing time
```

**Key rule**: Always use the ONTAP-generated timestamp (`event.timestamp`) as the canonical event time, not the ingest timestamp at the backend.

## Event Count Reconciliation

### Source vs Backend Comparison

```bash
# Step 1: Count source events (S3 audit files for a date range)
SOURCE_COUNT=$(aws s3api list-objects-v2 \
  --bucket <audit-bucket> \
  --prefix "audit/<svm-name>/2026/01/" \
  --query "Contents[].Key" --output json | jq length)

echo "Source files: ${SOURCE_COUNT}"

# Step 2: Count events per file (requires parsing)
# Use the log-parser layer to count events in each EVTX/XML file
python3 -c "
from fsxn_log_parser.parser import parse_audit_log
import json, sys
# Count events in a sample file
events = parse_audit_log('sample-audit.evtx')
print(f'Events in file: {len(events)}')
"

# Step 3: Compare with backend event count
# Datadog: Use Log Analytics query
# Grafana/Loki: Use LogQL count query
# Honeycomb: Use count() query
```

### Reconciliation Policy

| Frequency | Action | Acceptable Variance | Escalation |
|-----------|--------|--------------------:|------------|
| Daily (automated) | Compare S3 file count vs backend event count | ≤ 1% | Alert if exceeded |
| Weekly (manual) | Review discrepancy trends | ≤ 0.5% cumulative | Investigate root cause |
| Monthly | Full reconciliation report | 0% unexplained | Compliance team review |
| Post-incident | After Collector outage or config change | Document gap | Reprocess from DLQ/S3 |

### Common Discrepancy Causes

| Cause | Direction | Detection | Resolution |
|-------|-----------|-----------|------------|
| Lambda retry → duplicate | Backend > Source | Duplicate event_id | Deduplicate at backend |
| Collector outage → missing | Backend < Source | Gap in timeline | Reprocess from DLQ |
| Filter/sampling applied | Backend < Source | Expected by config | Document in reconciliation |
| Rotation timing | Temporary mismatch | Resolves within 1 cycle | Wait for next poll |

## Collector Config Change Audit Trail

### Why Track Config Changes

Every change to `otel-collector-config.yaml` affects which events reach which backends. For compliance, you must prove:
- What routing was active at any point in time
- Who approved the change
- Whether any events were lost during the change

### Required Audit Fields

| Field | Source | Example |
|-------|--------|---------|
| Who | Git author | `user <user@example.com>` |
| When | Git timestamp | `2026-01-15T10:30:00+09:00` |
| What | Git diff | Added `otlp_http/elastic` exporter |
| Why | PR description | "Add Elastic backend for security team" |
| Approval | PR reviewer | `security-team-lead` |
| Deployment | CI/CD timestamp | `2026-01-15T11:00:00+09:00` |

### Audit Query Commands

```bash
# Full history of Collector config changes
git log --format="%H %ai %an %s" -- \
  'integrations/otel-collector/otel-collector-config*.yaml' \
  > collector-config-audit.log

# Show what changed in a specific commit
git show <commit-hash> -- \
  'integrations/otel-collector/otel-collector-config*.yaml'

# Find who added/removed a specific exporter
git log -p --all -S 'otlp_http/elastic' -- \
  'integrations/otel-collector/otel-collector-config*.yaml'
```

### Config Change Impact Assessment

Before applying any config change, assess:

| Question | Risk if Skipped |
|----------|-----------------|
| Does this add/remove a backend? | Events may be lost or duplicated |
| Does this change filtering rules? | Compliance events may be dropped |
| Does this change batch settings? | Latency or throughput affected |
| Does this change retry settings? | Data loss window may increase |
| Is rollback tested? | Extended outage if change fails |
