# Enterprise Workload Addendum

🌐 [日本語](../ja/enterprise-workload-addendum.md) | **English** (this page)

## Applicability to SAP, VMware, and Mission-Critical Operations

This OTel Collector pattern applies when enterprise operations teams need to distribute FSx for ONTAP telemetry to multiple operational domains:

- Infrastructure / platform operations
- Security operations (SOC/SIEM)
- Application operations (SAP Basis, database administration)
- Managed service provider operations
- Observability vendor evaluation and migration

## What This Pattern Does

- Routes normalized audit events, EMS alerts, and FPolicy signals to multiple backends simultaneously
- Decouples the telemetry producer (Lambda) from backend-specific delivery
- Enables backend migration without application code changes
- Centralizes credential management and routing policy

## What This Pattern Does NOT Replace

- Raw audit evidence retention (EVTX/XML in S3 with Object Lock)
- Workload-native monitoring (SAP Solution Manager, Oracle Enterprise Manager, etc.)
- Application-level instrumentation (traces, metrics)
- Compliance evidence chain of custody

## FSx for ONTAP Workload Context

FSx for ONTAP supports enterprise workloads including:
- SAP HANA and SAP application servers
- Oracle Database and Oracle RAC
- Microsoft SQL Server
- VMware datastores (via NFS)
- General-purpose NAS (NFS/SMB)

The audit log, EMS, and FPolicy telemetry generated by these workloads can all be routed through the OTel Collector pattern described in this repository.

## Migration Approach for Enterprise Environments

For mission-critical workloads, use a phased approach:

1. **Parallel run**: Direct-send and Collector path operate simultaneously
2. **Parity validation**: Compare event count and critical fields for at least one full audit rotation cycle
3. **Operational sign-off**: Confirm backend queries, alerting, and runbooks work with Collector-delivered data
4. **Config freeze**: Lock Collector config before production cutover
5. **Cutover**: Disable direct-send path
6. **Observation period**: Monitor for 2+ weeks before removing old infrastructure
